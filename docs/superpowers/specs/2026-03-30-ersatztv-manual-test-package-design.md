# ErsatzTV 手测发布包与双击启动器设计

日期：2026-03-30

## 目标

为 Windows 手工测试提供一个“开箱即用”的交付物，避免直接双击 `bin\Debug\net10.0\ErsatzTV.exe` 时产生“缺少 Web 前端”或“不会自动打开界面”的误解。

目标体验：

- 用户拿到一个 zip 包
- 解压后双击启动器即可使用
- 启动器自动启动 ErsatzTV 服务
- 启动器自动打开浏览器到 `http://localhost:8409`
- 如果已有旧实例运行，则按用户要求执行“强制重启后再打开浏览器”
- 交付物不依赖测试机额外安装 .NET SDK / Runtime

## 非目标

本次不做以下内容：

- 不制作传统安装程序（MSI / Inno Setup / NSIS）
- 不把 ErsatzTV 改造成原生桌面 GUI 应用
- 不新增长期维护的原生 Launcher.exe 项目
- 不改变 ErsatzTV 主应用的 Web 架构或默认访问模式
- 不尝试解决所有生产级安装、服务注册、开机自启问题

## 选型结论

采用方案：**自包含 publish 文件夹 + 顶层双击启动器 + zip 打包**。

原因：

- 最符合“手测包”“双击即可”的短期目标
- 比 Debug 输出更接近真实交付物
- 比单文件 publish 更稳，排错更容易
- 比新增原生壳程序成本更低，风险更可控

## 交付物结构

最终产物为一个 zip，例如：

`ErsatzTV-manual-test-win-x64.zip`

解压后的目录结构：

- `启动 ErsatzTV.cmd`
- `README-手测说明.txt`
- `app\...`（ErsatzTV 自包含 publish 输出）
- `runtime\launcher-state.json`（启动器状态文件，首次启动后生成）
- `runtime\logs\...`（启动器日志，按需生成）

说明：

- `app` 目录保留主程序及其依赖，避免单文件发布带来的资源与排障问题
- `runtime` 目录用于保存启动器自己的状态和日志，避免污染 `app`

## 启动器行为

### 双击入口

用户双击 `启动 ErsatzTV.cmd` 后：

1. 调用同目录下的 PowerShell 启动逻辑
2. 校验 `app\ErsatzTV.exe` 是否存在
3. 读取上次启动记录（如存在）
4. 尝试结束旧实例
5. 启动新的 ErsatzTV 进程
6. 轮询 `http://localhost:8409`
7. 服务可访问后自动打开默认浏览器
8. 将新的进程信息写回状态文件

### 已有实例时的处理

按用户要求采用 **强制重启** 策略：

- 优先根据状态文件里的 PID 精准结束上次由该启动器启动的进程
- 如果 PID 不存在或已失效，则继续启动新实例
- 不默认粗暴结束系统上所有 `ErsatzTV.exe`，以避免误伤其他环境或手工启动的实例

如果后续证明旧实例识别不够稳，可增加有限兜底规则，但第一版优先采用“记录 PID + 精准结束”的保守方案。

## 默认地址与端口

第一版固定使用：

- UI 地址：`http://localhost:8409`

原因：

- 当前代码默认 UI 端口即为 8409
- 这是用户当前最直接的手测路径
- 避免启动器第一版引入额外配置复杂度

后续若需要，可扩展为从环境变量或配置文件读取端口，但这不是本次必要范围。

## 打包方式

### 主程序发布

采用 `win-x64`、`Release`、`self-contained` 发布。

发布目标：

- 让测试机无需额外安装 .NET Runtime
- 保留目录式发布而非单文件发布
- 产物直接落入手测包的 `app` 目录

### 启动器实现

第一版启动器使用：

- `启动 ErsatzTV.cmd` 作为双击入口
- PowerShell 脚本负责实际控制逻辑

理由：

- 实现速度快
- Windows 上双击兼容性好
- 便于调试和后续修改
- 满足“只需双击运行”的目标

## 日志与故障可见性

为降低手测时的黑盒感，启动器应提供最小可见性：

- 将启动器关键动作写入 `runtime\logs`
- 包括：时间、旧实例处理、启动命令、等待 URL、浏览器打开结果、异常信息

`README-手测说明.txt` 中应写明：

- 如何启动
- 默认访问地址
- 如果浏览器未自动打开，手动访问哪个 URL
- 如果启动失败，查看哪个日志文件

## 安全与边界

第一版启动器必须遵守：

- 不全局杀死所有同名进程
- 不修改系统级服务配置
- 不写注册表
- 不要求管理员权限
- 不侵入替换用户已有 ErsatzTV 安装

## 验收标准

完成后需要满足：

1. 能生成一个 zip 手测包
2. 测试机解压后无需额外安装 .NET 即可运行
3. 双击 `启动 ErsatzTV.cmd` 能启动 ErsatzTV
4. 浏览器会自动打开 `http://localhost:8409`
5. 第二次双击会执行“重启旧实例后再打开浏览器”
6. 用户不会再因为直接看到 exe 而误以为“缺少 web 端”
7. 出问题时能从 README 和启动器日志定位问题

## 测试策略

至少覆盖以下验证：

1. **冷启动**：无运行实例时双击启动器，服务成功启动并打开浏览器
2. **重复启动**：已有旧实例时再次双击，旧实例被替换，新实例成功运行
3. **手动浏览器兜底**：自动打开失败时，README 提供明确地址，手工访问可成功
4. **打包完整性**：zip 解压后目录结构完整、关键文件存在
5. **无 Runtime 依赖测试**：在不依赖开发机环境变量的前提下正常启动

## 风险与缓解

### 风险 1：PID 状态文件失真

表现：状态文件里的 PID 已不存在，或被系统复用。

缓解：

- 结束进程前先验证 PID 是否存在
- 第一版只做存在性校验，不做复杂的进程命令行匹配逻辑
- 如果 PID 无效，则跳过结束逻辑并直接启动新实例

### 风险 2：服务启动慢，浏览器打开过早

表现：浏览器打开时页面暂不可用。

缓解：

- 启动器轮询 `http://localhost:8409`
- 只有在探测成功后才打开浏览器
- 设置合理超时，并在超时时提示日志位置

### 风险 3：目录式发布体积较大

表现：zip 包偏大。

缓解：

- 接受该成本，优先稳定性与可排错性
- 本次手测阶段不为了体积改成高风险单文件方案
